## 1. Общее описание

**JACK NUTRITION** — интернет-магазин биологически активных добавок (БАДов), построенный по архитектуре «статический фронтенд + Supabase backend».

### Бизнес-логика
1. **Витрина** — покупатель открывает сайт, просматривает каталог товаров, фильтрует, ищет, добавляет в корзину.
2. **Корзина** — хранится в браузере (`localStorage`), содержит только `id` товара и `количество`.
3. **Оформление** — при нажатии «Оформить заказ» фронтенд отправляет корзину в Supabase Edge Function `create-order`.
4. **Защита цен** — Edge Function самостоятельно проверяет товары в базе, берёт актуальные цены, считает итог, формирует текст заказа и открывает WhatsApp.
5. **Аналитика** — анонимная запись о продаже сохраняется в таблицу `orders_analytics` (без данных клиента).

### Ключевые ограничения
- **Нет следов о посетителях**: нет кук, нет логирования IP, нет таблиц с персональными данными.
- **Нет серверной части для фронтенда**: статика на GitHub Pages.
- **Все расчёты цен на сервере**: клиент не может подменить итог.

### Связи
- **Frontend ↔ Supabase REST API**: читает товары, категории, бренды, настройки через анонимный ключ.
- **Frontend ↔ Edge Function `create-order`**: отправляет `{ cart }`, получает `whatsappUrl` и номер заказа.
- **Frontend ↔ Edge Function `admin-api`**: отправляет JWT от Supabase Auth, получает/отправляет данные админки.
- **Edge Functions ↔ PostgreSQL**: используют `service_role` ключ для обхода RLS и записи в защищённые таблицы.
- **Storage**: публичная для чтения, только для аутентифицированных — для записи.

## 4. Edge Functions

### 4.1 `create-order` — обработка заказа

**URL**: `https://hpphfeojjejculvdundj.supabase.co/functions/v1/create-order`

**Метод**: `POST`

**Входные данные**:
```json
{
  "cart": [
    { "id": "uuid-товара", "quantity": 2 }
  ]
}
```

**Логика**:
1. Проверяет, что корзина не пуста.
2. Читает настройки из таблицы `settings`.
3. **Проверяет время оформления**: если включён лимит, проверяет текущий час в указанном часовом поясе (`timezone`). Если нерабочее время — возвращает `403` с `time_restricted: true`.
4. Запрашивает все товары из корзины из БД (только `id`, `name`, `price`, `stock`, `is_visible`).
5. **Подставляет актуальные цены** — игнорирует любые подделки от клиента.
6. Ограничивает количество `stock` (не продаёт больше, чем есть на складе).
7. Пропускает невидимые или несуществующие товары, ставит флаг `hasError` и спецкод `order_error_code`.
8. Получает следующий номер заказа из `order_counter` (формат `ДДММГГ/XXX`).
9. Увеличивает счётчик.
10. Сохраняет запись в `orders_analytics` (анонимная).
11. Формирует текст заказа для WhatsApp.
12. Возвращает `whatsappUrl` для открытия.

**Успешный ответ** (`200`):
```json
{
  "success": true,
  "orderNumber": "130726/137",
  "total": 5000,
  "items": [
    { "name": "Товар 1", "quantity": 2, "price": 2500, "total": 5000 }
  ],
  "whatsappUrl": "https://wa.me/79991234567?text=...",
  "hasError": false,
  "errorCode": ""
}
```

**Ошибки**:
- `400` — корзина пуста / нет доступных товаров
- `403` — нерабочее время (`time_restricted: true`)
- `500` — ошибка БД, ошибка генерации номера

**Особенности**:
- Использует `SUPABASE_SERVICE_ROLE_KEY` для обхода RLS.
- CORS: разрешены запросы с любого origin (`req.headers.get("origin")`), в том числе с GitHub Pages.
- При ошибке аналитики/счётчика заказ всё равно проходит (логируется в консоль).
- Имеет публичный эндпоинт `GET /health` для проверки работоспособности.

### 4.2 `admin-api` — единый API для админки

**URL**: `https://hpphfeojjejculvdundj.supabase.co/functions/v1/admin-api`

**Аутентификация**: JWT-токен из Supabase Auth в заголовке `Authorization: Bearer <token>`. Проверяет, что email пользователя есть в таблице `admin_users`.

**Поддерживаемые маршруты**:

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/settings` | Получить все настройки как `{ key: value }` |
| `PUT` | `/settings` | Сохранить настройки (upsert по `key`), пустые значения нормализуются в `null` |
| `GET` | `/products?search=&category=&brand=&page=&limit=` | Список товаров с пагинацией, поиском по `name`, `sku`, `barcode` |
| `POST` | `/products` | Создать товар (с images/links) |
| `PUT` | `/products/:id` | Обновить товар (с images/links, полная замена) |
| `DELETE` | `/products/:id` | Удалить товар (каскадное удаление images/links) |
| `GET` | `/categories` | Список категорий |
| `POST` | `/categories` | Создать категорию |
| `PUT` | `/categories/:id` | Переименовать категорию |
| `DELETE` | `/categories/:id` | Удалить категорию |
| `GET` | `/brands` | Список брендов |
| `POST` | `/brands` | Создать бренд |
| `PUT` | `/brands/:id` | Переименовать бренд |
| `DELETE` | `/brands/:id` | Удалить бренд |
| `GET` | `/analytics?period=` | Статистика за период (`day`, `week`, `month`, `quarter`, `year`, `all`) |
| `GET` | `/orders?period=&page=&limit=` | Список заказов с пагинацией |
| `POST` | `/import` | Импорт товаров из Excel (JSON-массив) |
| `GET` | `/export` | Экспорт всех товаров (JSON) |
| `GET` | `/backup` | Дамп всех таблиц (JSON-файл) |
| `GET` | `/health` | Проверка работы функции |

**Особенности**:
- Все ответы JSON с CORS-заголовками.
- При импорте автоматически создаёт категории/бренды, если их нет.
- При обновлении по `sku` пустые поля Excel не затирают существующие значения.
- При ошибках импорта возвращает `{ row, error }` для каждой строки.
- `export` возвращает JSON, на фронтенде клиентская библиотека `xlsx` преобразует его в `.xlsx`.
- Ошибки возвращаются в формате `{ error: "текст" }` с соответствующим HTTP-статусом.

### Логика фронтенда

#### Инициализация (`init`)
1. Применяет тему из `localStorage`.
2. Загружает настройки из `admin-api/settings`.
3. Загружает товары через Supabase REST API (`/rest/v1/products?is_visible=eq.true`).
4. Загружает категории и бренды для фильтров.
5. Рендерит каталог.
6. Обновляет счётчик корзины.
7. Настраивает обработчики событий.
8. Проверяет ограничение по времени (`checkOrderTime`), запускает интервал на 60 сек.

#### Поиск и фильтры
- Поиск по `name` и `brands.name` (клиентская фильтрация).
- Фильтры по `category_id`, `brand_id`, диапазону `price`.
- Сортировка: `price-asc`, `price-desc`, `name`, `newest` (по `created_at`).
- Debounce для инпутов: 300 мс (поиск), 500 мс (цена).

#### Корзина
- Хранилище: `localStorage` (ключ `jack-cart`), только `{ id, quantity }`.
- При изменении перерендеривает каталог (чтобы обновить кнопки на карточках).
- Счётчик в шапке: сумма quantity всех товаров.

#### Оформление заказа (`checkout`)
1. Блокирует кнопку, показывает «Оформление...».
2. `POST` на `create-order` с телом `{ cart }`.
3. Если `response.ok` — открывает `whatsappUrl` в новой вкладке, очищает корзину, показывает alert.
4. Если `time_restricted` — показывает сообщение в модалке.
5. Если ошибка — alert.
6. В `finally` разблокирует кнопку.

#### Ограничение по времени (`checkOrderTime`)
- Запрашивает `admin-api/settings`.
- Если `order_time_limit_enabled === 'false'` — кнопка активна.
- Иначе проверяет текущий час в часовом поясе `timezone` (через `Intl.DateTimeFormat`).
- Если нерабочее время: кнопка `disabled = true`, показывает текст.

#### Сканер штрих-кодов
- Использует `BarcodeDetector` API (только Chrome/Edge).
- Запрашивает доступ к камере (`facingMode: 'environment'`).
- В цикле `requestAnimationFrame` детектирует штрих-коды.
- При обнаружении — ищет товар по `barcode` через REST API, открывает модалку.

- **Категории/Бренды**: таблица с редактированием названия через кастомное модальное окно `#nameModal`.
- **Аналитика**: Chart.js график выручки, топ-10 товаров, список заказов.
- **Настройки**: форма с полями магазина, часовой пояс, лимит времени, спецкод, валюта; ошибки сохраняются явно.
- **Импорт**: `XLSX.read` (SheetJS), отправляет JSON на `admin-api/import`.
- **Экспорт**: `XLSX.writeFile` из JSON `admin-api/export`.
- **Бэкап**: скачивание JSON-дампа.
- **Мониторинг**: polling `create-order/health` каждые 30 сек, индикатор (зелёный/серый кружок), WhatsApp-уведомление при падении.

## 6. ДИЗАЙН И UX (Универсальная адаптивность + iOS-стиль на мобильных)
Проект проектируется как кросс-платформенное веб-приложение, которое одинаково хорошо выглядит и работает на любом устройстве: iPhone, Android, планшеты, ноутбуки, настольные ПК и даже экраны с нестандартным разрешением (вплоть до 320px по ширине). Основной принцип — mobile-first, но с полной адаптацией под все размеры экранов. На мобильных устройствах сайт визуально имитирует нативное iOS-приложение, следуя Apple Human Interface Guidelines (HIG). На десктопах и планшетах сохраняется та же логика, но компоновка элементов меняется для комфортного использования с мышью.

7.1. Основополагающие Принципы Дизайна
Отзывчивость (Responsive): Все элементы подстраиваются под ширину экрана. Нет горизонтального скролла. Верстка «резиновая», с использованием %, vw, clamp() и медиа-запросов.

Единый UX: На всех устройствах сохраняется логика навигации и взаимодействия, но меняется компоновка (например, на десктопе больше колонок в каталоге, на мобильных — нижний таб-бар).

iOS-стиль на мобильных: На телефонах (ширина < 768px) интерфейс максимально приближен к нативному iOS: системные шрифты, большие зоны касания (44pt), стандартные паттерны (bottom sheet, таб-бар, навигация свайпом).

Доступность (Accessibility): Минимальный размер интерактивных элементов — 44x44pt для пальцев (на мобильных), на десктопе можно меньше. Все кнопки имеют понятные подписи.

Производительность: Лёгкая анимация (только CSS-трансформации и opacity), не нагружающая CPU. Изображения оптимизированы.

Без лишнего: Поскольку у нас нет авторизации, профиля и настроек пользователя, они не выводятся в интерфейсе. Вся навигация строится вокруг трёх основных разделов: каталог, корзина, поиск.

7.2. Макет и Сетка
Гибкая сетка (CSS Grid + Flexbox)

Каталог товаров строится на CSS Grid с автоматическим заполнением:

css
.catalog {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(160px, 1fr));
  gap: 16px;
}
На телефонах (ширина < 480px) — 2 колонки.

На небольших телефонах (320–374px) — 2 колонки, но карточки чуть уже (адаптивно через minmax(140px, 1fr)).

На планшетах (480–1024px) — 3–4 колонки.

На десктопах (>1024px) — 4–5 колонок, с ограничением максимальной ширины контейнера в 1200px.

На широких мониторах (>1600px) — до 6 колонок, но контент центрируется.

Отступы (Margins)

На мобильных: 16px по краям экрана.

На планшетах и десктопах: 24–32px, чтобы контент не растягивался на всю ширину.

Высота элементов

Карточка товара на мобильных: динамическая, подстраивается под контент.

Кнопки в карточке: фиксированная высота 44px (на всех устройствах).

7.3. Навигация
На мобильных устройствах (ширина < 768px)

Используется нижний таб-бар (как в iOS) с тремя пунктами (без настроек и профиля):

Каталог (иконка дома) — главный экран.
Корзина (иконка корзины с бейджем количества товаров).
Поиск (иконка лупы) — при нажатии открывается поле поиска поверх контента.
Верхняя панель: логотип (слева) + переключатель тёмной/светлой темы (справа).

На планшетах и десктопах (>768px)

Нижний таб-бар преобразуется в верхнюю панель навигации (горизонтальное меню) или остаётся внизу (если так удобнее пользователю). Для единообразия можно оставить нижний таб-бар на всех экранах — это допустимо и для веба.

Дополнительно в шапке появляется строка поиска (всегда видна).

7.4. Типографика и Цвета
Шрифты

Используются системные шрифты для максимальной производительности и родственного ощущения на каждом устройстве:

css
font-family: -apple-system, BlinkMacSystemFont, "SF Pro Display", "SF Pro Text", "Segoe UI", Roboto, Oxygen, Ubuntu, Cantarell, sans-serif;
Размеры текста задаются в относительных единицах (rem, clamp), чтобы они масштабировались на больших экранах:

css
h1 { font-size: clamp(1.8rem, 5vw, 3rem); }
.price { font-size: clamp(1rem, 2vw, 1.5rem); }
.product-name { font-size: clamp(0.9rem, 1.5vw, 1.2rem); }
iOS-стиль на мобильных (подражание SF Pro):

Large Title: 34pt жирный.

Headline: 17pt полужирный.

Body: 17pt обычный.

Caption: 12pt.

Цветовая схема

Используются системные цвета, которые автоматически адаптируются под светлую и тёмную тему:

Основной акцент (Tint Color): зелёный (#34C759) для кнопок «В корзину» и «Оформить». На тёмной теме — чуть ярче.

Фоны: #F2F2F7 (светлая тема), #000000 или #1C1C1E (тёмная).

Текст: #000000 (светлая), #FFFFFF (тёмная).

Второстепенный текст: #8E8E93 (светлая), #AEAEB2 (тёмная).

Переключатель темы

Иконка солнца/луны в правом верхнем углу.

При переключении меняются CSS-переменные (через класс .dark-theme или атрибут data-theme).

7.5. Компоненты и Взаимодействие
Карточка товара (в каталоге)

На мобильных: фото сверху, под ним бренд (мелко), название, форма выпуска, цена, кнопка «В корзину» (или счётчик). Всё в 2 колонки.

На десктопе: такая же вертикальная карточка, но с более крупными элементами.

При клике на карточку (любую область, кроме кнопки) открывается модальное окно с деталями.

Кнопка «В корзину» (и счётчик)

Состояние «Добавить»: большая круглая кнопка с иконкой корзины или текст «В корзину».

После добавления: кнопка заменяется на блок «–» (уменьшить), счётчик (количество), «+» (увеличить).

Все три элемента имеют область нажатия ≥44x44pt (на мобильных).

Модальное окно (детали товара)

На мобильных: выезжает снизу (bottom sheet), занимает ~80% экрана. Закрывается свайпом вниз или кнопкой «Закрыть».

На десктопе: центрированное окно (popup) с затемнённым фоном, закрывается кликом вне окна или по крестику.

Содержит: полное описание, состав, дозировку, способ применения, противопоказания, фото в большом размере.

Поиск

На мобильных: иконка лупы в таб-баре → при нажатии открывается поле ввода поверх контента (с анимацией).

На десктопе: поле ввода всегда видно в шапке.

Поиск по штрих-коду: отдельная иконка рядом с полем → при нажатии открывается камера для сканирования.

Корзина

Отображается как отдельная вкладка в таб-баре.

Список товаров с кнопками +/–, удалением (свайп или крестик), общей суммой.

Кнопка «Оформить заказ» — большая, зелёная, внизу экрана (липкая).

7.6. Адаптивные изображения
Для фото товаров: используется атрибут srcset или тег <picture> для загрузки разного размера в зависимости от экрана (экономия трафика и ускорение загрузки).

Пример:

html
<img src="photo-small.jpg" srcset="photo-medium.jpg 600w, photo-large.jpg 1200w" sizes="(max-width: 480px) 160px, (max-width: 768px) 200px, 250px" alt="...">
Для фонов: background-size: cover + медиа-запросы для замены картинок.

7.7. Интерактивность и Обратная связь
На мобильных: касание → лёгкая вибрация (navigator.vibrate) при успешном добавлении в корзину или ошибке. Визуальное изменение кнопки (цвет, масштаб).

На десктопе: hover-эффекты (подсветка, масштабирование), курсор pointer.

Анимации: только CSS-трансформации и opacity (не анимировать высоту/ширину, чтобы не было тормозов).

Сообщения об ошибках: системный тост (снизу) или стандартный alert (пока).

7.8. Адаптивность для нестандартных экранов
Экраны холодильников (≈1024x600): используется та же верстка, что и для планшетов (3–4 колонки). Шрифты и кнопки достаточно крупные.

Ультраширокие мониторы (21:9): контент центрируется, максимальная ширина контейнера 1400px.

Мобильные с очень маленьким экраном (320px): отступы уменьшаются до 8px, шрифты сжимаются до минимального размера (но не менее 14px для читаемости).

7.9. Тестирование адаптивности
Инструменты: Chrome DevTools (эмуляция устройств), BrowserStack (для реальных устройств).

Минимальная поддерживаемая ширина: 320px (iPhone SE).

Критические брейкпоинты: 320px, 480px, 768px, 1024px, 1200px, 1600px.

Регулярная проверка: после каждого изменения, особенно после добавления новых компонентов.

Как это применить в коде
В frontend/styles.css уже должны быть:

CSS-переменные для цветов и шрифтов.

Медиа-запросы для всех брейкпоинтов.

Использование clamp() для размеров шрифтов и отступов.

В index.html — использование семантических тегов и атрибутов srcset для изображений.

В app.js — добавление navigator.vibrate() в функции добавления в корзину и обработки ошибок.

## 7. Безопасность и анонимность

### Защита цен
- Фронтенд отправляет только `{ id, quantity }`.
- Edge Function `create-order` самостоятельно запрашивает цены из БД (`price`, `stock`, `is_visible`).
- Любые подделки цен/количеств от клиента игнорируются.
- Если товар не найден или не видим — в WhatsApp добавляется спецкод (`[!CHECK!]`).

### Анонимность
- **Нет кук** для идентификации посетителей.
- **Нет логирования IP** (Supabase не хранит IP по умолчанию, логи отключены).
- **Нет таблиц** с персональными данными клиентов (имя, телефон, город, email).
- **Аналитика** (`orders_analytics`) содержит только: номер заказа, список товаров, сумма, дата.

### Ключи
| Ключ | Где хранится | Для чего |
|------|--------------|----------|
| `SUPABASE_URL` | Env переменная Edge Function | Подключение к БД |
| `SUPABASE_SERVICE_ROLE_KEY` | Env переменная Edge Function | Полный доступ к БД (обход RLS) |
| `anon key` | Вшит в `app.js` и `admin/app.js` | Публичный REST API (чтение) |
| `admin-token` | `localStorage` браузера | JWT для админки (после логина) |

**Важно**: `anon key` — это публичный ключ проекта, его можно использовать в клиентском коде. Секретный ключ (`service_role`) нигде в репозитории не хранится.

### RLS
- Публичный доступ: только чтение видимых товаров, категорий, брендов, настроек, изображений.
- Защищённые таблицы (`orders_analytics`, `order_counter`, `admin_users`): доступ только через `service_role`.
- Запись/удаление: только через Edge Functions с `service_role`.

## 10. Инструкция по запуску (для нового разработчика)

### Предварительные требования
- Аккаунт на GitHub.
- Аккаунт на Supabase (бесплатный план).
- Node.js (для Supabase CLI).
- Git.

### Шаг 1: Клонировать репозиторий
```bash
git clone https://github.com/Jabraiil/JOCK-NUTRITION.git
cd JOCK-NUTRITION
```

### Шаг 2: Создать проект Supabase
1. Зайдите на [supabase.com](https://supabase.com), создайте проект.
2. В **SQL Editor** выполните содержимое файла `supabase/migrations/001_initial_schema.sql`.
3. При необходимости связанных товаров и SQL-дампа выполните `supabase/migrations/002_related_products_and_sql_backup.sql` (создаёт таблицу `product_related`, RLS и функцию `generate_sql_dump()`).
3. В **Storage** создайте bucket `product-images` (public, до 5MB, image/*).
4. В **Table Editor → admin_users** добавьте свой email.

### Шаг 3: Развернуть Edge Functions
```bash
npm install -g supabase
supabase login
supabase init
supabase link --project-ref <ваш-project-id>
supabase functions deploy create-order --no-verify-jwt
supabase functions deploy admin-api --no-verify-jwt
```
Скопируйте URL функций:
- `https://<project-id>.supabase.co/functions/v1/create-order`
- `https://<project-id>.supabase.co/functions/v1/admin-api`

### Шаг 4: Настроить фронтенд
Откройте `app.js`, найдите `CONFIG` и проверьте, что там стоят реальные значения вашего проекта:
```javascript
const CONFIG = {
    supabaseUrl: 'https://hpphfeojjejculvdundj.supabase.co',
    supabaseAnonKey: '...',
    orderFunctionUrl: 'https://hpphfeojjejculvdundj.supabase.co/functions/v1/create-order',
    adminApiUrl: 'https://hpphfeojjejculvdundj.supabase.co/functions/v1/admin-api'
}
```

### Шаг 5: Настроить админку
В `admin/app.js` аналогично проверьте `CONFIG`.

### Шаг 6: Включить GitHub Pages
1. Settings → Pages в репозитории.

## 11. Дополнительные замечания

- **GitHub Pages** обслуживает статику из корня репозитория. В репозитории нет папки `docs/` как источника сайта.
